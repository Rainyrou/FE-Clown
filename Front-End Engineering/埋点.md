埋点即在应用内的用户交互和生命周期等关键节点内嵌采集代码，将用户行为转换为量化指标，核心价值为技术 → 业务 → 商业闭环，数据流架构为前端采集层 → 传输层 → 存储计算层 → 应用层，验证功能迭代效果 + 定位异常交互场景 + 优化应用性能体验 + 优化推荐算法策略 + 构建用户兴趣画像 + 监控业务链路转化

| 埋点模式       | 底层原理                                                                       | 适用场景                   | 优点               | 缺点                                                   |
| -------------- | ------------------------------------------------------------------------------ | -------------------------- | ------------------ | ------------------------------------------------------ |
| 业务自定义埋点 | 研发在业务代码中显式调用埋点 SDK 的 `track` 方法，传入事件名和自定义属性       | 核心业务链路、复杂属性采集 | 数据精准、属性灵活 | 侵入业务代码、开发成本较高、迭代较慢、容易出现漏埋错埋 |
| 全埋点         | 埋点 SDK 在根节点监听所有事件和组件生命周期并自动采集                          | 探索性分析、快速迭代       | 无侵入性、全覆盖   | 数据冗余、性能开销大                                   |
| 可视化埋点     | 通过可视化平台监听页面元素，配置埋点事件名和自定义属性，生成配置文件下发给 SDK | 运营/PM 自行配置、快速迭代 | 无代码             | 兼容性差                                               |

混合式方案：全埋点为基础 + 业务自定义埋点为核心
事件机制：

- 命名规范：业务线-页面-组件-事件
- 支持同步/异步注册 `track(eventName, properties, callback)`
* 事件类型：行为事件（click/touch/scroll） + 生命周期事件（page_show/page_hide/app_start/app_exit） + 业务事件（order_create/pay_success/business_switch） + 异常事件（error_js/api_fail/crash）

属性设计：所有埋点默认携带基础属性，业务埋点补充自定义属性
评审流程：PM 提出埋点需求 → 埋点设计 → 技术评审 → 埋点字典录入 → 前端实现 → 测试验证 → 灰度发布 → 全量上线
业务链路：`app_start`（APP 启动）→ `business_show`（业务曝光）→ `business_click`（业务点击）→ `order_create`（下单）→ `order_pay`（支付）→ `order_complete`（订单完成）→ `order_repeat`（复购）
