系统设计：

- 没人能在短时间内完美回答问题的所有细节，实际工作中也不可能在迭代开发中深度钻研每一个底层原理 ，但大厂问这些问题的核心，从来不是考你是否全懂，而是判断你是否真的主导项目、是否有落地思考、是否能抓住核心逻辑
* 面试核心并非"完美"，是"有逻辑 + 有落地细节"，你是主导者而非执行者，能说清"为什么选这个方案""排除了哪些选项""解决了什么具体问题"；你懂"核心原理"而非"只会用工具"，能讲清方案的核心逻辑，无需深入底层源码，比如Gecko 预拉取是提前存储资源到本地，绕开网络"；你有"避坑意识"和"量化思维"，能提到落地时的风险（如流量消耗、兼容性）和最终结果（如 TTI 降 10%）
* 如何快速回应高难度问题？以"Gecko 预拉取的底层存储机制"为例，无需硬编，根据这一逻辑回答：

> "Gecko 离线包的底层存储我没深究（坦诚边界），但我聚焦在业务侧的策略设计：它是将资源包和 template.js 存在本地缓存（核心逻辑），我通过生服提供的 JSB 实现上游页面预拉取，比常规 Channel Group 粒度更细 —— 比如用户在车系页停留超过 2 秒，且点击报价单概率高时才触发，避免无效拉取（落地细节）。落地时我考虑了两个风险：弱网下预拉取失败则降级为 CDN 加载，资源过期则通过版本号静默更新（避坑意识），最终让 TTI 缩短 10%，首屏加载率提升 20%（量化结果）"

* "迭代快没时间思考"本质是将"底层原理"和"落地细节"混为一谈，实际工作中可以不钻研"Gecko 的存储机制"，但必须思考"预拉取会不会耗流量""Tree Shaking 失效怎么解决"，这些都是"不思考就会出问题"的落地细节，而非"额外思考"

举例"Tree Shaking 为什么失效，你如何解决"

❌ 啰嗦版："Tree Shaking 是 webpack 的功能，用来删除未使用的代码，Rspack 和 webpack 原理类似，class 之所以有副作用，是因为 Rspack 的副作用判定规则里，class 的构造函数可能有执行逻辑，比如静态属性、实例化时的操作，所以会被标记为有副作用，Tree Shaking 就失效了，然后我改了 bam.config.json，把 exportClassService 设为 false，开启 exportFunction，这样产物就变成 function 了，function 没有这些副作用，Tree Shaking 就恢复了，然后还遇到了 lodash 和 lodash-es 共存的问题，又写了插件…"

✅ 精简版（1 分钟内）：

* 结论：Tree Shaking 失效是因为 bam 默认生成 class，被 Rspack 判断有副作用，无法移除未使用代码
* 核心逻辑：Rspack 对 class 和 function 的副作用判断不同 ——function 更易被判断为"无副作用"，因此核心是将产物从 class 转换为 function
* 落地细节（+ 实习亮点）：我修改了 bam 的配置，同时参考字节实习时处理依赖冗余的经验，发现还有 lodash 和 lodash-es 共存的问题，所以自定义 Rspack 插件，在 AST 层面直接替换引用，既解决 Tree Shaking，又减少 300KB 包体积；
* 避坑点：一开始没注意到 lodash 的重复依赖，后来通过 Rsdoctor 查依赖图谱才发现，所以重构时一定要先进行依赖分析
